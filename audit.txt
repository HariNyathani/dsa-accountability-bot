## 🔴 1. Critical: UserMenu calls `useNavigate()` after an early return (Rules of Hooks violation)

**File:** `frontend/src/components/UserMenu.tsx:40`

```ts
export default function UserMenu() {
  const { user, authenticated, loading, login, logout, isAdmin } = useAuth();

  if (loading) return ( <div className={s.loading}…>…</div> );            // early return
  if (!authenticated || !user) return ( <div className={s.wrap}>…</div> ); // early return

  const initials = user.username.slice(0, 2).toUpperCase();
  const navigate = useNavigate();   // ← called only on the "authenticated" branch
```

React requires hooks to be called in the **same order on every render**. Here `useNavigate` is called only after two early returns. On the first render (loading) React sees 1 hook; on the next render (authenticated) React sees 2 hooks — **"Rendered fewer hooks than expected" runtime error in dev**, and the user menu will break the entire tree the moment auth resolves from `loading→authenticated`.

**Fix:** Hoist `useNavigate` above the early returns.
```ts
export default function UserMenu() {
  const { user, authenticated, loading, login, logout, isAdmin } = useAuth();
  const navigate = useNavigate();
  if (loading) return …;
  if (!authenticated || !user) return …;
  // …
}
```

## 🔴 2. Critical: Login UI is unreachable on mobile-web (sidebar-only UserMenu)

**File:** `frontend/src/components/Layout.tsx:97` + `frontend/src/components/Layout.module.css:169-176`

## 🔴 3. Critical: Admin users table uses raw `<a href>` (full page reload, not a Router Link)

**File:** `frontend/src/pages/AdminPage.tsx:195`

```tsx
<a href={`/users/${u.user_id}`} className={sh.username} style={{ color: "var(--link)" }}>
  {u.discord_username || "Unknown"}
</a>
```

A plain anchor in an SPA causes a **full document reload** — drops in-memory state, re-runs the entire `AuthProvider`/`ThemeProvider` bootstrap (FOUC flash), re-fires the no-FOUC script, loses the active LiquidCapsule selection, and re-fetches every endpoint. The rest of the app uses React Router everywhere.

**Fix:** Import and use `<Link>`:
```tsx
import { Link } from "react-router-dom";
// …
<Link to={`/u/${u.username || u.user_id}`} className={sh.username}>{u.discord_username || "Unknown"}</Link>
```
(prefer the vanity URL to match the other "profile" links in the app).

## 🟠 4. High: Leaderboard table rows use `<tr onClick>` (not keyboard-accessible, not semantic)

**File:** `frontend/src/pages/LeaderboardPage.tsx:73`

```tsx
<tr key={e.user_id} onClick={() => nav(`/u/${e.username || e.user_id}`)}>
```

A `<tr>` is **not focusable**, is not announced as interactive to screen readers, and is not reachable by Tab. Keyboard users and assistive-tech users cannot navigate. This is also a known React anti-pattern (non-button element with click handler).

**Fix:** Move the navigation onto the username cell as a `<Link>` (semantic, focusable, right-clickable, screen-reader announces "link, X"), and remove the `<tr onClick>`. Optionally also keep a row-level onClick *plus* `tabIndex={0}` and `onKeyDown` for Enter/Space — but the cleanest, most accessible fix is the `<Link>`.

## 🟠 5. High: UserMenu (and any custom-trigger Dropdown) is not keyboard-operable

**File:** `frontend/src/components/Dropdown.tsx:108-122` and `frontend/src/components/UserMenu.tsx:66`

The custom trigger path renders:
```tsx
<div role="button" aria-haspopup="menu" aria-expanded={open} data-open={open}
     onClick={() => setOpen((o) => !o)} style={{ display: "contents" }}>
  {children}
</div>
```

Two compounding problems:
1. **`display: contents`** removes the element from the box tree entirely — there is no focusable surface, no tab-order entry, and `role="button"` on a box-less element is not focusable by Tab.
2. The custom child is a `<div>` (UserMenu) — also not focusable, no `onKeyDown` for Enter/Space.

**Net effect:** The account menu cannot be opened with a keyboard; the trigger is invisible to the focus ring; only mouse works.

**Fix:** Either
- Render the custom trigger as a real `<button>` (the `trigger="trigger"` path already does), with the user-supplied children inside it, **or**
- Keep the wrapper but require the custom child to be an accessible element (or pass `forwardRef`/clone) and add `onKeyDown` for Enter/Space that calls `setOpen(o => !o)`.

Minimum patch: in the custom path, attach the keyboard handler to the wrapper:
```tsx
<div role="button" tabIndex={0} aria-haspopup="menu" aria-expanded={open}
     onClick={() => setOpen(o => !o)}
     onKeyDown={(e) => { if (e.key === "Enter" || e.key === " ") { e.preventDefault(); setOpen(o => !o); }}}
     style={{ display: "contents" }}>
```
And drop `display: contents` (which makes the role/focus invisible) — or replace the whole wrapper with a `<button type="button" className={s.field}>` and let the child fill it.

## 🟡 8. Medium: `useApi` `deps` parameter is `unknown[]` — no compile-time help, easy to forget

**File:** `frontend/src/hooks/useApi.ts:17,50`

```ts
deps: unknown[] = [],
// …
}, [tick, enabled, ...deps]);
```

Callers pass `[uid]`, `[period]`, `[sortBy]`, etc. — but TS can't catch a missing dep. Across the codebase, the spread is correct (every page passes the right dependency), but the loose typing means a future caller who forgets the dep will silently get stale fetches.

**Fix:** Tighten to `DependencyList` (from React) and document. Bonus: switch the effect to use `useRef` for the latest `fetcher` so the dep can be `[tick, enabled, ...deps]` without an eslint-disable.

## 🟡 9. Medium: Vite emits a single 877KB chunk (chunk-size warning) — no `manualChunks`

**File:** `frontend/vite.config.ts`

The bundle is 877KB unminified / 259KB gzip; Vite explicitly warns `chunks > 500kB`. Recharts + framer-motion are the two biggest deps. Splitting the vendor chunk (recharts, motion) into its own file would let the browser cache it across deploys and let code-split routes (MyDashboard, Analytics, Admin, RevisionBank) defer loading the chart library until needed.

**Fix:**
```ts
// vite.config.ts
import { defineConfig } from "vite";
// …
build: {
  rollupOptions: {
    output: {
      manualChunks: {
        react: ["react", "react-dom", "react-router-dom"],
        motion: ["motion/react"],
        recharts: ["recharts"],
      },
    },
  },
  chunkSizeWarningLimit: 800,
},
```
This drops the warning and shortens first-paint JS for routes that don't need charts.

## 🟡 10. Medium: `AdminRoute` and `AdminPage` async effects have no `cancelled` flag

**Files:** `frontend/src/components/AdminRoute.tsx` (no async effect), `frontend/src/pages/AdminPage.tsx:71-74`

```ts
useEffect(() => {
  fetchUsers();
  api.health().then((r) => setHealthData(r.data)).catch(() => {});
}, [fetchUsers]);
```

If the admin user navigates away mid-fetch, the `.then` callback calls `setHealthData` on an unmounted component → React logs a warning and leaks the closure. Easy to fix with a `cancelled` ref or `AbortController`.

**Fix:**
```ts
useEffect(() => {
  let cancelled = false;
  fetchUsers();
  api.health().then((r) => { if (!cancelled) setHealthData(r.data); }).catch(() => {});
  return () => { cancelled = true; };
}, [fetchUsers]);
```

## 🟡 11. Medium: `useApi`'s exhaustive-deps is disabled — `fetcher` closure not in deps

**File:** `frontend/src/hooks/useApi.ts:49-50`

```ts
return () => { cancelled = true; };
// eslint-disable-next-line react-hooks/exhaustive-deps
}, [tick, enabled, ...deps]);
```

The current callers all spread the *real* deps (`[uid]`, `[period]`, `[sortBy]`) so the effect re-fires when needed and the `fetcher` closure is fresh. But this is fragile — a new caller who passes `[]` (empty deps) when they should pass `[id]` will get a stale fetcher forever. The eslint-disable hides the smell rather than fixing it.

**Fix:** Keep the latest `fetcher` in a ref:
```ts
const fetcherRef = useRef(fetcher);
useEffect(() => { fetcherRef.current = fetcher; });
// then inside the effect: fetcherRef.current()
```
Now the effect can depend on `[tick, enabled, ...deps]` without lint, and is always safe.

## 🟡 12. Medium: `Heatmap` recomputes a stale 365-day range (no re-eval past midnight) + 730 inline arrow handlers

**File:** `frontend/src/components/Heatmap.tsx:22-49, 105-106`

Two real issues:
1. The `useMemo([], …)` builds the 365-day window **once**. The first user to open the page at 23:59 sees a range that doesn't include tomorrow's "active day" until they hard-reload. Minor, but a `useMemo([todayKey], …)` (where `todayKey` is `formatDate(new Date())`) would re-anchor on midnight.
2. The 365 tiles each create fresh `onMouseEnter`/`onMouseLeave` closures on every render (~730 new function objects per re-render). Either use event delegation on the parent `.cells` (one handler that finds the target via `e.target.dataset`), or memoize tile components. Minor perf, but the scale (365 × 730) makes it the most render-heavy thing in the app.

## 🟡 13. Medium: `Dropdown` panel `min-width: 100%` is broken under `display: contents` custom triggers

**File:** `frontend/src/components/Dropdown.module.css:71` + `Dropdown.tsx`

The trigger path is:
- `trigger="trigger"` → renders a real `<button>` inside a `.field` (`position: relative`) — the panel anchors correctly via `min-width: 100%`.
- `trigger="custom"` → the trigger wrapper uses `display: contents`, so `.field` has **no positioned box** (the wrapper's children participate in the *grandparent* layout, not `.field`). The panel, which is `position: absolute; top/bottom: calc(100%+10px) / calc(100%-8px)`, anchors to `.field` — but `.field` is now collapsed/inline with `display: contents`, so the panel's `top/left: 0` resolves to a nearby box's coordinate and the panel appears in the wrong place or at `(0,0)`.

**Fix:** Either
- Apply `position: relative` to a stable ancestor in the custom-trigger case (e.g., a real wrapper div with no `display: contents`), or
- Don't use `display: contents`; use `width: 100%; display: flex` so `.field` keeps its positioned box and the children fill it.

## 🟡 15. Medium: `useApi` initial `data=null` + `loading=true` means pages with `if (overview.error) return` short-circuit on first paint, hiding the skeleton

**File:** `frontend/src/pages/AnalyticsPage.tsx:32` (and similar in DashboardPage, MyDashboardPage, UserProfilePage)

```ts
if (overview.error) return <ErrorState … />;
const o = overview.data;
```

This is fine **after** the first fetch resolves. But on the first render `overview.error` is `null` and `overview.loading` is `true` and `overview.data` is `null`. The code proceeds to `const o = overview.data;` (null) and the JSX checks `overview.loading ? <Skeleton/> : o && <grid>` — correct. So this is OK, **but** if `overview` ever errors *during* a refetch (while `o` is still the previous data), `o` holds stale data and we render `<grid>` with stale content while the error boundary is **skipped because the error check runs first** → we render stale. The right pattern: render skeleton when `loading && !data`; render `ErrorState` on error **or** on `error && !data`; show stale data with a non-blocking toast on `error && data`. Minor.
