# Frontend Audit Report — Post-Redesign Diagnosis

**Date:** Fri Jun 26 2026
**Auditor:** opencode (minimax-m3)
**Scope:** `frontend/src/` — React 19 + TS 5.8 + Vite 6.4
**Build:** `tsc --noEmit` ✅ | `vite build` ✅ → 925 KB JS (260 KB gzip) across **5 chunks** (react 49 KB, motion 130 KB, recharts 421 KB, index 276 KB, CSS 49 KB). No Vite warning at `chunkSizeWarningLimit: 600`.

---

## Executive Summary

The redesign landed most of the critical and high-severity fixes from the prior audit. The Rules-of-Hooks violation, the `<a href>` full-reload in Admin, the Leaderboard `<tr onClick>` anti-pattern, the Dropdown `display: contents` positioning bug, the stale 365-day Heatmap, the manualChunks warning, and the duplicate `Skeleton`/`Loader` exports are all genuinely resolved. **The app is now buildable, type-clean, and most user-visible regressions are gone.**

However, the mobile-login fix is **incomplete** (ThemeSwitcher is still sidebar-only on mobile), three prior bugs are **not fixed** (AdminPage `<Select>` value mismatch, `useRevisionBank` ↔ `useApi` inconsistency, Dropdown listbox arrow-key nav), and the redesign introduced **a new crop of bugs** — most importantly: no `ErrorBoundary` (any render error blanks the app), no skip-to-content link, duplicate `<UserMenu/>` mount on mobile, Modal `onClose` instability causing body-overflow flicker, and the RevisionBank row click pattern repeating the same Leaderboard anti-pattern that was just fixed elsewhere.

**Confidence in the fixes:** high. **Confidence in the codebase being production-ready:** medium — the new issues are individually small but collectively represent ~20 outstanding defects across 8 categories.

## New Bugs Found

### NEW-1 — 🟠 High — Mobile cannot switch theme (regression of original bug #2)
**File:** `src/components/Layout.tsx:99, 136-138`, `src/components/Layout.module.css:186-200`
The mobile-auth fix from the prior audit only added `<UserMenu/>` to the mobile auth strip. `<ThemeSwitcher/>` remains mounted only inside the sidebar (line 99), which is `display: none` at `max-width: 900px` (`Layout.module.css:171`). The mobile media query (lines 190-200) only reveals the `.mobileAuth` div — it does not duplicate the theme switcher. Result: a mobile user can log in, but cannot toggle light/dark/system.
**Recommended fix:** render a compact `<ThemeSwitcher compact />` inside `.mobileAuth` (it already accepts a `compact` prop at `ThemeSwitcher.tsx:16`).

### NEW-2 — 🟡 Medium — Duplicate `<UserMenu/>` mount on mobile
**File:** `src/components/Layout.tsx:103, 137`
On every page render, `<UserMenu/>` is mounted **twice**: once in the sidebar (line 103, `display: none` on mobile but still in the React tree) and once in `.mobileAuth` (line 137, only `display: block` on mobile). Each instance maintains its own `Dropdown` state, so opening the menu in the mobile chip while the sidebar instance is also mounted means two popovers, two focus traps, and two `useEffect` subscriptions to the same `useAuth()` context. Wasteful, and a future contributor who adds a global keyboard shortcut will accidentally wire it to both.
**Recommended fix:** conditionally render only the relevant instance based on a viewport media query, or use a single shared `<UserMenu/>` mounted at the layout root with a CSS-only swap of which container is visible.

### NEW-3 — 🟡 Medium — `Modal` body-overflow flicker on parent re-render
**File:** `src/components/Modal.tsx:30-39`, called from `src/pages/RevisionBankPage.tsx:220`
The `useEffect` that locks `document.body.style.overflow` and binds the `Esc` handler has `[open, onClose]` as its dep array. `RevisionBankPage.tsx:220` passes `onClose={() => setSelectedProblem(null)}` — a new function on every render. Each parent re-render while the modal is open: cleanup fires (sets `body.style.overflow = ""`), then the effect re-fires (sets it back to `"hidden"`), causing a single-frame body-scroll flicker. Also: the `Esc` listener is detached and re-attached on every render.
**Recommended fix:** in `RevisionBankPage.tsx`, wrap with `useCallback`:
```ts
const handleCloseModal = useCallback(() => setSelectedProblem(null), []);
// <Modal onClose={handleCloseModal} ... />
```

### NEW-4 — 🟡 Medium — `AdminPage` `<Select>` controls have no `label` prop
**File:** `src/pages/AdminPage.tsx:241, 265, 294, 328`
The `Select` component from `Field.tsx` is used for `sudoUserId`, `sudoDiff`, `restUserId`, and `undoUserId` — none pass a `label` prop. The `Field.tsx:34` only renders a `<span className={s.label}>{label}</span>` if `label` is truthy, so the user sees a bare `<select>` with no visible label and no `htmlFor` association. Screen readers will not announce a name. The same form is rendered with the section title (`⚡ Sudo-Log — Insert Progress Entry`) as the only contextual label, which is not associated.
**Recommended fix:** pass `label="Target user"` / `label="Difficulty"` etc. to each `<Select>`.

### NEW-5 — 🟡 Medium — `RevisionBankPage` `<div onClick>` row not keyboard accessible
**File:** `src/pages/RevisionBankPage.tsx:35-71` (line 44 specifically)
The `ProblemRow` component renders `<div className={s.row} onClick={() => onOpen(item)}>`. This is the **same anti-pattern that the prior audit flagged and fixed in `LeaderboardPage.tsx:73`**: non-button element with click handler, not focusable, not announced as interactive, not reachable by Tab. Keyboard-only and screen-reader users cannot open the modal. The audit fixed this in one place; the codebase has at least one other copy of the same bug.
**Recommended fix:** add `role="button"`, `tabIndex={0}`, and `onKeyDown` for Enter/Space — or convert to a real `<button>`.

### NEW-6 — 🟡 Medium — `submitReview` triggers 2× `fetchDueItems`
**File:** `src/hooks/useRevisionBank.ts:66-78`, `src/pages/RevisionBankPage.tsx:97-102`
`useRevisionBank.submitReview` (line 72) calls `await fetchDueItems()` internally after a successful review. Then `RevisionBankPage.handleReview` (line 100-101) calls `await fetchPagedHistory(1, limit)` and `await fetchDueItems()` again. Net: one user review → 3 API calls (`submitRevisionReview` + `fetchDueItems` × 2). The second `fetchDueItems` overwrites the first; the first is wasted bandwidth and a race against the second.
**Recommended fix:** remove the `fetchDueItems` call from `useRevisionBank.submitReview` and let the caller orchestrate the refresh sequence.

### NEW-7 — 🟡 Medium — Heatmap data is invisible to screen readers
**File:** `src/components/Heatmap.tsx:147-155`
The floating tooltip is wrapped in `aria-hidden="true"` (line 151) — correct, since it's a visual-only element. But there is **no alternative accessible representation** of the 365-day data. The tile `<div>`s have no `role`, no `tabIndex`, no `aria-label`. A screen-reader user navigating the Heatmap card gets no information about which days have activity, what the streak is (the visual "Active Days: N" stat on line 81 is visible but the per-day data is not exposed). WCAG 1.1.1 (Non-text Content) is violated for this data visualization.
**Recommended fix:** add an off-screen `<table>` (visually hidden, `class="sr-only"` from `base.css:106-116`) summarizing the heatmap data, or add `aria-label` to each tile with the date + count.

### NEW-8 — 🟡 Medium — `MyDashboardPage` fires 6 parallel API calls on mount
**File:** `src/pages/MyDashboardPage.tsx:37-42`
Six `useApi` hooks (`userInfo`, `aggregate`, `sums`, `remind`, `activity`, `heatmap`) all start with `loading: true` and all fire their fetches simultaneously on mount. The backend (FastAPI on localhost:8000 / Render) receives 6 concurrent requests from the same authenticated user. Same pattern in `UserProfilePage.tsx:33-37` (5 calls).
**Recommended fix:** add a shared `dashboardAggregate` endpoint (one round-trip), or stagger the calls with a short delay / `requestIdleCallback`.

### NEW-9 — 🟡 Medium — `useApi` does not abort in-flight `fetch` on dep change
**File:** `src/hooks/useApi.ts:38-62`
The `cancelled` flag (line 46) prevents the `.then` callback from calling `setData` on an unmounted/stale component, but the underlying `fetch` request continues to completion. On a rapidly-changing dep (e.g., `period` toggling in `AnalyticsPage.tsx:30`), 4-5 parallel in-flight requests can be active simultaneously, all hitting the same endpoint. The browser limits concurrent requests to the same origin (6 per Chrome), so this can cause queueing.
**Recommended fix:** pass an `AbortController` signal into the fetcher:
```ts
const ctl = new AbortController();
fetcherRef.current({ signal: ctl.signal })...
return () => { cancelled = true; ctl.abort(); };
```
(Requires the fetcher closures to accept and forward the signal.)

### NEW-10 — 🟡 Medium — No React `ErrorBoundary` anywhere
**Files:** `src/App.tsx`, `src/main.tsx`, `src/components/*.tsx`
A search for `ErrorBoundary` / `componentDidCatch` returned zero matches. An unhandled render error in any page (e.g., a malformed `aggregate.data` from the backend, a Recharts crash on a new chart type) will unmount the entire React tree and show a blank page. The Vite dev server would show an overlay, but in production the user sees nothing.
**Recommended fix:** add a minimal `ErrorBoundary` class component, mount it around `<Outlet />` in `Layout.tsx` (line 116-120), and render a glass-styled "Something went wrong — reload" card on `componentDidCatch`.

### NEW-11 — 🟡 Medium — `QuickLogCard` success/error feedback is not announced
**File:** `src/components/QuickLogCard.tsx:168-182`
The feedback `<motion.div>` (line 170) has no `role="status"` and no `aria-live="polite"`. The emoji is correctly hidden from screen readers (fix #17), but the message itself — `"✅ Successfully logged 5 arrays questions!"` — is also not announced because the element is not in a live region. WCAG 4.1.3 (Status Messages) is violated.
**Recommended fix:** add `role="status"` and `aria-live="polite"` to the motion.div (or wrap the message in a `<span aria-live="polite">`).

### NEW-12 — 🟡 Medium — `Dropdown` custom-trigger path drops `ariaLabel`
**File:** `src/components/Dropdown.tsx:92, 109-126`
The `trigger="trigger"` button (line 87-107) sets `aria-label={ariaLabel}` (line 92). The `trigger="custom"` wrapper (line 109-126) does **not** apply the `ariaLabel` prop. The wrapper has `role="button"` and `aria-haspopup="menu"` but no accessible name. In `UserMenu.tsx:58`, `ariaLabel="Account"` is passed to the `Dropdown` but never reaches the custom trigger. The visible text content (avatar + name + badge + chev) provides an implicit name, but if a future caller passes textless children, the button is unnamed.
**Recommended fix:** add `aria-label={ariaLabel}` to the custom-trigger `<div>` (line 109).

### NEW-13 — 🟡 Medium — `Modal` dialog has no `aria-labelledby`
**File:** `src/components/Modal.tsx:93-114`
The dialog has `role="dialog"`, `aria-modal="true"` (lines 102-103), but no `aria-labelledby` linking to the title `<h3>` (line 107). Screen readers will not announce the dialog title when the modal opens. The `<h3>` is rendered conditionally (line 105 — only when `title` prop is truthy), so adding `aria-labelledby` is conditional too.
**Recommended fix:** generate a `useId()` for the title, set `id={titleId}` on the `<h3>` and `aria-labelledby={titleId}` on the dialog.

### NEW-14 — 🟡 Medium — `Modal` close button is hidden when no `title` is passed
**File:** `src/components/Modal.tsx:105-112`
The close button is inside the `{title && (...)}` block. If a future caller passes a Modal without a title (currently no caller does — `RevisionBankPage.tsx:221` always passes a title — but the type allows `title?: ReactNode`), the user has no in-modal close button and must click the backdrop or press `Esc`. The backdrop click works (line 91), but the missing visible button is a discoverability regression.
**Recommended fix:** always render the close button; only the `<h3>` title is conditional.

### NEW-15 — 🟡 Medium — `UserProfilePage` `validId` regex too restrictive
**File:** `src/pages/UserProfilePage.tsx:31`
```ts
const validId = /^\d{17,21}$/.test(uid) || /^[a-z0-9_]{4,20}$/.test(uid);
```
Discord snowflake IDs are typically 17-19 digits (not 21). The upper bound of 21 is over-permissive and silently allows 20-21 digit IDs that don't exist. More importantly: the `/users/:userId` route (registered in `App.tsx:25`) was kept for backward compatibility, but the regex may reject older IDs that don't match the new 17-21 digit range.
**Recommended fix:** tighten to `/^\d{17,19}$/` for the snowflake half.

### NEW-16 — 🟡 Medium — Mobile auth chip truncates username aggressively
**File:** `src/components/Layout.module.css:198`
```css
.mobileAuth { max-width: 180px; }
```
The UserMenu trigger inside the chip contains: a 36 px avatar, the `.name` span (with `white-space: nowrap; text-overflow: ellipsis`), the `.badge` ("Authenticated"), the `.info` wrapper, and the `.chev` (12 px). With 12-16 px gaps, the name gets ~80 px of width before truncation. For a username like "harinyathani", the rendered text will be "harin…" — visually broken.
**Recommended fix:** bump to `max-width: 240px` on the chip, or hide `.name`/`.badge` on small screens via a media query and show only the avatar + chev.

### NEW-17 — 🟡 Medium — No skip-to-content link
**File:** `src/components/Layout.tsx:46-141`
A search for `skip` / `skipToMain` / `skip-link` returned zero matches. Keyboard-only users tabbing through the page must traverse the entire sidebar nav + theme switcher + user menu + footer on every page load before reaching `<main>`. WCAG 2.4.1 (Bypass Blocks) is violated.
**Recommended fix:** add `<a href="#main" className="sr-only">Skip to main content</a>` (with `:focus` style to make it visible) as the first child of `.shell` in `Layout.tsx:75`, and add `id="main"` to the `<main>` element (line 116).

### NEW-18 — 🟡 Medium — `tokens.css` `--heat-0` light still faint at 0.18
**File:** `src/styles/tokens.css:57`
`--heat-0: rgba(122, 82, 52, 0.18)` on the Alabaster background `#F2E2DE` produces a barely-visible darkening — effectively the "no activity" tiles blend into the background, defeating the heatmap's purpose. The prior audit recommended bumping from 0.1 to a more visible value; 0.18 is a marginal improvement but still inadequate. (Computed contrast against `#F2E2DE` is ~1.3:1, well below the 3:1 WCAG large-element minimum.)
**Recommended fix:** bump to `0.25` or `0.30`, or use a separate solid token like `#E0CFC8` (a desaturated chestnut ~10% darker than the background).

### NEW-19 — 🟡 Medium — `useRevisionBank` has no abort on unmount
**File:** `src/hooks/useRevisionBank.ts:32-78`
The three async functions (`fetchDueItems`, `fetchPagedHistory`, `submitReview`) call `setLoading` / `setError` / `setDueItems` in their `finally` blocks with no `cancelled` flag. If the user navigates away from `/revision` while a fetch is in flight, the state setters fire on an unmounted component. React 18+ doesn't warn, but it's a real leak. The companion `useApi` hook has a proper `cancelled` flag (`useApi.ts:46`).
**Recommended fix:** add a `cancelled` ref + cleanup, or migrate the hook to use `useApi` internally per the prior audit's recommendation (bug #28).

### NEW-20 — 🟡 Medium — `RevisionBankPage` doesn't reset state on tab switch
**File:** `src/pages/RevisionBankPage.tsx:91-95`
The `useEffect` fires `fetchDueItems` / `fetchPagedHistory` when the tab changes, but the previous tab's state (`dueItems`, `allRevisionItems`, `topicStats`) is not cleared. Switching from "All Problems" (where `allRevisionItems` has 10 items) to "Due Today" (where `dueItems` is empty) shows the "Loading queue…" message correctly (line 134) but the moment the new fetch resolves with empty `dueItems`, the user sees the "Queue Cleared!" cheer — correct. However, if the user switches back to "All Problems" before the previous fetch resolves, the displayed list is whichever fetch completed last. Minor race.
**Recommended fix:** when `currentTab` changes, reset the other lists to `[]` so the UI can't show stale data from a different tab.

### NEW-21 — 🟢 Low — `Dsalogo.png` not WebP, not optimized
**File:** `public/Dsalogo.png` (56 KB)
The logo is a 56 KB PNG served as-is. No WebP alternative, no `<picture>` element, no `srcset`. For an above-the-fold asset on every page, this is the LCP candidate. Modern formats would cut this to ~15-20 KB.
**Recommended fix:** convert to WebP + AVIF, use `<picture>` with the PNG fallback, or use `vite-imagetools` for build-time optimization.

### NEW-22 — 🟢 Low — `Heatmap` re-defines `formatDate` and `weekdays` on every render
**File:** `src/components/Heatmap.tsx:50, 53-58`

### NEW-23 — 🟢 Low — `useApi` error messages are browser-specific technical text
**File:** `src/hooks/useApi.ts:54-55`
On a network failure (server down, CORS, offline), `err.message` is `"Failed to fetch"` (Chrome/Edge) or `"NetworkError when attempting to fetch resource"` (Firefox) or `"Load failed"` (Safari). The `ErrorState` component shows this raw text. A user sees "Load failed" and doesn't know if it's their connection or the server.
**Recommended fix:** wrap network errors in `useApi` and produce a human-friendly message: `"Couldn't reach the server. Check your connection and retry."`

### NEW-29 — 🟢 Low — `UserMenu` initials calculation unsafe
**File:** `src/components/UserMenu.tsx:40`
```ts
const initials = user.username.slice(0, 2).toUpperCase();
```
The `AuthUser.username` field is typed as `string` (required) in `AuthContext.tsx:20`, so the TypeScript type is correct. But the API can in practice return a user with `username: null` or `""` (the type is a contract, not a runtime guarantee). On `""`, `initials` becomes `""` (renders an empty fallback div); on `null`, `null.slice(0, 2)` throws.
**Recommended fix:** `const initials = (user.username ?? "??").slice(0, 2).toUpperCase();`
