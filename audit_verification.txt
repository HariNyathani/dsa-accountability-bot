[1] Daemon-thread API server co-located with Discord bot
    Severity: Medium | Audit says: main.py:41-64, 101-105
    Status: VERIFIED
    Confidence: High
    Evidence: main.py:54-62 — start_api_thread() creates a

    start_api_thread() then start_bot() which blocks.  The daemon flag

[5] ADMIN_DISCORD_ID is str but BOT_OWNER_ID is int
    Severity: Low | Audit says: config.py:67
    Status: VERIFIED
    Confidence: High
    Evidence: config.py:26 — BOT_OWNER_ID is int via _safe_int().
    config.py:67 — ADMIN_DISCORD_ID is str via os.getenv(), defaulting to

[26] Connection pool lazily initializes in get_connection
    Severity: Medium | Audit says: db/database.py:32-49
    Status: VERIFIED
    Confidence: High
    Evidence: db/database.py:39-49 — get_connection() checks
    `if not self.pool` and calls init_pool().  init_pool() (line 30-36)
    has `if self.pool: return` — this is not atomic.  Two threads could
    both pass the check, creating two pool instances.  The first pool
    would be leaked (overwritten by self.pool = SimpleConnectionPool(...)).
    The API startup in app.py:32 calls db_manager.init_pool() during
    lifespan, which mitigates this for the API path.  But the bot path
    calls database.init_db() in on_ready, which doesn't call init_pool().
    The race condition is real but unlikely to manifest in the default
    single-process deployment.

[27] All DB calls use asyncio.to_thread with sync psycopg2
    Severity: Medium | Audit says: db/database.py:53-54
    Status: VERIFIED
