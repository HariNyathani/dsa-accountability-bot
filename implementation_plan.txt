MODULE 1 — DATABASE CONCURRENCY: CONSTRAINTS & CONNECTION POOL
Covers: P3-02 (TOCTOU limit races, schema side), P3-04 (pool exhaustion / thread
safety). CRIT/HIGH. This is the FOUNDATION — do it first.
================================================================================

GOAL
Make the database enforce the invariants that application code currently races
on, and replace the non-thread-safe / non-blocking connection pool so that a
burst of concurrent requests cannot crash the whole app.

TARGET FILES
- db/database.py                       (pool construction, ~lines 22, 36, 43, 51)
- init_postgres.sql                    (add constraints / partial indexes)
- db/migrations/003_concurrency_constraints.sql   (NEW FILE)

AGENT INSTRUCTIONS
1. CONNECTION POOL (P3-04) in db/database.py:
   a. Change the import from:
          from psycopg2.pool import SimpleConnectionPool
      to:
          from psycopg2.pool import ThreadedConnectionPool
   b. Replace BOTH occurrences of
          SimpleConnectionPool(5, 25, self.dsn)
      (in init_pool() and in get_connection()) with:
          ThreadedConnectionPool(5, 50, self.dsn)
      i.e. minconn=5, maxconn=50. Read the max from an env var with a default:
          minconn = int(os.getenv("DB_POOL_MIN", "5"))
          maxconn = int(os.getenv("DB_POOL_MAX", "50"))
          ThreadedConnectionPool(minconn, maxconn, self.dsn)
   c. Wrap the getconn() call inside get_connection() in a bounded retry loop so
      that a transient "connection pool exhausted" PoolError waits and retries
      instead of 500-ing immediately. Example:
          from psycopg2.pool import PoolError
          import time
          conn = None
          for attempt in range(10):          # ~ up to 2s total
              try:
                  conn = self.pool.getconn()
                  break
              except PoolError:
                  time.sleep(0.2)
          if conn is None:
              raise RuntimeError("DB pool exhausted after retries")
      Keep the existing `with conn:` / finally putconn(conn) structure intact.

2. NEW MIGRATION FILE db/migrations/003_concurrency_constraints.sql (P3-02):
   Add DB-level guards. Use IF NOT EXISTS / partial unique indexes so the
   migration is idempotent and does not blow up on existing duplicate data
   (if duplicates exist, the CREATE will fail — see step 4).
       -- One rest day per user per calendar day
       CREATE UNIQUE INDEX IF NOT EXISTS idx_progress_one_rest_per_day
           ON progress_logs (user_id, log_date)
           WHERE message_type = 'rest';
   NOTE: daily_status already has UNIQUE(user_id, date) — do NOT re-add it.

3. Mirror the same CREATE UNIQUE INDEX statement into init_postgres.sql so fresh
   installs get it too. Place it after the progress_logs table definition.

4. DE-DUPLICATION GUARD: Before the CREATE UNIQUE INDEX in the migration, add a
   cleanup that collapses any pre-existing duplicate rest rows so the index can
   be built:
       DELETE FROM progress_logs a
       USING progress_logs b
       WHERE a.message_type = 'rest' AND b.message_type = 'rest'
         AND a.user_id = b.user_id AND a.log_date = b.log_date
         AND a.id > b.id;

VERIFICATION STEP
- Static: grep db/database.py — confirm ZERO occurrences of "SimpleConnectionPool"
  remain and "ThreadedConnectionPool" appears; confirm PoolError retry loop present.
- DB: run the migration against a scratch DB; confirm it applies cleanly and

- Functional: insert two 'rest' rows for the same (user_id, log_date) directly

- Regression: start the API, hit any existing GET endpoint that reads the DB

MODULE 9 — BEARER-ONLY AUTH ENFORCEMENT (CSRF NOT APPLICABLE)
Covers: P2-04 (originally "no CSRF tokens"). DOWNGRADED — see rationale.
================================================================================

ARCHITECTURE DECISION (authoritative)
The Flutter mobile app authenticates STRICTLY via `Authorization: Bearer <token>`
headers. It does NOT use session cookies. Classic CSRF requires the browser to
auto-attach ambient credentials (cookies) to a cross-site request; a Bearer token
lives in app-controlled storage and is never auto-attached by a browser, so a
malicious third-party site CANNOT forge an authenticated request against this API.
=> CSRF is NOT an applicable threat vector for this architecture. Finding P2-04 is
   RECLASSIFIED as "not applicable" PROVIDED the API accepts credentials ONLY via
   the Authorization header. This module's job is to VERIFY and LOCK IN that
   precondition — NOT to build double-submit tokens or CSRF cookies.

DO NOT IMPLEMENT: CSRF token cookies, double-submit tokens, X-CSRF-Token headers,
or any per-request CSRF middleware. They add complexity with no security benefit
for a bearer-only API.

GOAL
Guarantee the API has exactly ONE authentication transport (Authorization: Bearer)
so the "CSRF-not-applicable" assumption cannot silently regress into a cookie
fallback that WOULD be CSRF-exploitable.

TARGET FILES
- api/middleware/auth.py               (token extraction logic)
